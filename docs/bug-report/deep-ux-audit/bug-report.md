# 深度用户体验审计：卡顿、预览污染与漏扫

## 报告来源

- 用户反馈：搜索、排序、翻页和预览存在持续卡顿；用 PPT Doctor 预览后，同一 PowerPoint 进程中的正常文档会变糊，并出现随机字母数字命名的隐藏演示文稿；部分目录未进入全盘索引。
- 审计方法：代码调用链、红绿单测、真实 PowerPoint COM 小样、Windows 进程/句柄取证、长时间 Qt/SQLite 压力测试。
- 安全边界：真实 COM 验证前确认没有用户 PowerPoint 进程；只处理测试创建且以创建时进程句柄证明归属的无窗口、零文档进程，不关闭用户文档或用户 PowerPoint。

## 已确认严重问题与根因

### 1. 隐藏 PowerPoint 预览进程污染后续正常打开

**复现**：渲染结束后调用 `renderer.shutdown()`，清理最长约 61 秒；此前隐藏 PowerPoint 实例可能继续存活，任务栏悬停可见哈希命名的临时快照。只退出 PPT Doctor 不能恢复，关闭全部 PowerPoint 后恢复正常。

**根因**：无界面 PowerPoint 的 `Application.HWND` 在动态 COM 下可能表现为调用成员并抛出“成员不存在”，导致晚期 PID 归属验证失败；代码因此不敢结束自己创建的无头实例。PowerPoint 单实例复用又把预览时的 DPI/临时文稿状态带给用户后续双击打开的文件。

**修复**：创建预览实例时立即记录唯一新进程 PID 和进程句柄；退出时先关本次打开的演示文稿，再用创建时句柄验证原进程仍存活、无可见窗口且 `Presentations.Count == 0`，最后执行有界退出。任一归属或可见性条件不满足就拒绝终止。

### 2. 一次性后台任务结束后不销毁，越用越卡

**复现**：长稳进程约 100 分钟后有 10,161 个 `Semaphore` 句柄；线程数只有 4。胶片报告等功能每次完成后都从 Python 任务列表移除，但已结束的 `QThread` 仍留作主窗口子对象。

**根因**：Qt 父子关系不会因 Python 列表 `remove()` 自动销毁对象；Windows 上每个残留线程对象继续占有一组内核同步句柄。

**修复**：所有 `BackgroundTask` 在调用方注册完结果/清理回调后，再把回收回调注册为最后一个 `finished` 槽；结束时先脱离长寿命父窗口，等待 1 秒让 Qt/Python 队列彻底排空，再在 GUI 所属线程 `deleteLater()`。此修复覆盖胶片报告、版本历史、诊断、导出、最近文件等全部一次性任务入口。

**修复过程中的安全门禁**：直接把 `finished` 连到 `deleteLater()` 虽能释放句柄，却在设置页连续测试中稳定触发 `Qt6Widgets.dll / 0xc0000005`。该方案已撤销；当前“最后注册 + 脱离父对象 + 1 秒宽限”的版本通过 30 个设置页用例、30 次真实设置对话框生命周期和 100 次胶片报告生命周期验证，没有再出现访问冲突。

### 3. 预览缓存命中检查会在 UI 线程访问原文件

**复现**：点击位于离线盘、断开的网络盘或云占位目录中的搜索结果，界面可能在真正渲染开始前就冻结。

**根因**：UI 线程调用 `find_cached_render()` 时隐式执行源文件 `os.stat()`；本地 SSD 上很快，网络/云路径却可能长时间阻塞，因此常规性能测试不容易暴露。

**修复**：UI 使用索引库已有的 `mtime + size` 生成缓存键，只查询本地预览缓存；需要重新核验源文件的操作仍在渲染工作线程执行。

### 4. “全盘扫描”错误排除普通业务目录 `Temp/tmp`

**复现**：`Projects\Temp\deck.pptx` 不会进入索引，即使目录并非系统临时目录、也不是新建目录。

**根因**：扫描器按目录名全局排除 `Temp` 和 `tmp`，把普通项目目录误判为系统缓存。

**修复**：仅排除相对于扫描根目录的真实 `AppData\Local\Temp` 子树，普通业务 `Temp/tmp` 恢复扫描；扫描策略版本升至 3，旧安装会安排一次覆盖性重扫。

### 5. 不可读目录会让全盘对账误删正常索引

**复现**：公司电脑权限、杀软临时拦截或离线目录导致遍历失败时，扫描结果看起来像“文件已删除”，旧索引可能被清掉。

**根因**：旧实现没有区分“本轮没看到”和“文件已确认不存在”，也没有把目录读取错误作为不完整扫描状态传播。

**修复**：记录不可读目录和样例路径；扫描不完整时保留旧记录，不执行基于缺失集合的删除；状态栏明确提示覆盖不完整。

### 6. 搜索连接失败或数据库维护锁会让界面长时间显示“正在搜索”

**复现**：启动时数据库路径短暂失败，搜索线程直接退出；之后每次请求都无人消费，界面一直等待。即使线程没有退出，只要索引库被维护/结构锁占用，首次搜索连接也会执行写向的 `PRAGMA journal_mode=WAL` 并沿用长等待策略；独立临时库实测约 7.4 秒才返回锁错误。

**根因**：SQLite 连接在线程主循环之外一次性创建，异常没有回传 UI，也没有给后续请求重试机会；交互搜索还错误复用了写连接初始化和 8 秒级 busy 等待。FTS 召回层会吞掉 `locked/busy` 后继续尝试其他查询，让多个等待窗口串起来。

**修复**：连接改为按请求延迟建立；搜索专用连接使用真正只读 URI 和 400ms busy timeout，不再执行写向 PRAGMA；`locked/busy` 只等待一次并上抛，界面约 0.9 秒内结束“正在搜索”、保留旧结果并用大白话提示稍后重试。线程继续存活，新查询会中断旧查询，陈旧结果不再覆盖当前界面。公司环境可能把 AppData 重定向到 UNC，URI 因此专门使用 SQLite 可接受的 `file:////server/share/...`，不调用可能触网的 `Path.resolve()`，并覆盖空格、`#`、`&` 与 Win32 扩展路径前缀。

### 7. 安全释放预览会话后，连续翻页过早再次冷启动

**复现**：原方案只预取一张相邻页并在约 0.35 秒空闲后释放隐藏 PowerPoint；第 2 页缓存命中，但第 3 页又需要约 2.1 秒重新启动 COM。安全问题解决了，翻页体验却仍呈周期性卡顿。

**根因**：预取上限固定为 1，没有充分利用当前已经打开、且已证明由 PPT Doctor 独占的短命渲染会话。

**修复**：当前页一显示，就以 960px、低优先级串行预取同方向连续 3 页。仍只有一个 COM 导出在执行，新预览会在每页之间抢占并清掉剩余队列；预取完成后照常释放隐藏 PowerPoint，不延长污染窗口。

### 8. 渲染清理卡住时可能叠加第二个隐藏 PowerPoint

**复现**：握手完成的渲染子进程被中断后，父进程只断开通信；若下一次预览立即重启渲染器，旧子进程仍可能在 PowerPoint COM 中清理，两个会话相互干扰。极端情况下，系统连清理等待线程都无法创建，旧计数还会永久卡住。

**根因**：原安全修复只避免“杀 Python 留 PowerPoint”，没有把“清理中的真实进程对象”作为下一次启动的门禁；门禁又依赖辅助线程递减计数，线程创建失败无法自愈。

**修复**：父进程保存每个待清理子进程的 `Popen` 对象；每次启动前用非阻塞 `poll()` 清理已退出项。只要旧子进程仍活着，就拒绝启动第二个渲染器；辅助线程构造/启动失败也不丢进程对象，下一次请求会自动恢复。真正卡死时保持“失败关闭”，绝不按超时粗暴杀 Python 子进程，因为后者会重新制造孤儿 `POWERPNT.EXE`。

### 9. 版本后端启动失败却继续显示“已开启”

**复现**：版本库损坏、权限不足或迁移失败时，设置和托盘入口仍显示版本保护已开启；之后每次保存都会重复触发快照失败与 watcher 重试。

**根因**：异步 `manager.start()` 失败只记录错误，没有回滚持久化开关和 UI；若失败发生在部分启动之后，已创建的维护资源也可能残留。正常退出路径还会重复调用两次 `stop()`。

**修复**：失败时由生命周期线程先清理半启动后端，再把版本开关持久化回关闭，通过 Qt 队列信号同步托盘、主窗和仍打开的设置对话框；正常退出只由生命周期线程清理一次，只有线程 3 秒后仍未退出才走兜底清理。

## 最终验证证据

- PowerPoint 真实 6 页翻页样例：第 1 页约 2.128 秒，第 2 页缓存/复用约 0.166 秒；释放约 0.567 秒；再次打开第 3 页约 2.141 秒，最终释放约 0.458 秒；测试后没有残留 `POWERPNT.EXE`。
- 修复前同一路径清理约 61 秒；修复后真实清理均低于 0.6 秒。
- 胶片报告生命周期实测：连续打开/关闭 100 次，进程句柄 282 → 289，遗留 `BackgroundTask` 为 0。
- 通用后台任务回归：500 个一次性任务结束后，Qt 子任务为 0；修复前红测保留全部 8/8 子线程对象。
- 真实 6 页 PPT 连续预取：首张冷启动约 2.019 秒；后台预取后续 3 页合计约 0.321 秒；第 2/3/4 页实际点击显示分别约 3.8/3.4/3.5 毫秒，失败数 0，结束后 `POWERPNT.EXE` 为 0。
- 普通业务 `Temp` 与系统 `AppData\Local\Temp` 均有针对性扫描测试，前者收录、后者排除。
- UI 预览缓存测试禁止调用源文件 `stat()`，确保离线路径不会卡住界面线程。
- 旧实现长时红灯基线自然跑满 21,200.031 秒（5.889 小时）：60,198 次操作、23,415 次搜索、9,850 次预览请求，业务错误 0；但 RSS 从 106.8MB 增至 435.2MB。在 15,180 秒监控点句柄已达 25,644，说明它不是立刻崩，而是悄悄越用越重。该进程启动于最终资源修复之前，只作为泄漏基线，不把其 UI 峰值当作修复版表现。
- 最终代码无并发 15 分钟高频压测：2,560 次操作、999 次搜索、290 次预览请求、错误 0；操作 p95 18.91ms、最大 75.38ms；界面独立监控最大间隙 79ms、慢间隙 0；搜索 p95 12.07ms、最大 14.68ms，active/pending 均为 0；末分钟 CPU 0%，最终 RSS 125.1MB。
- 数据库独占锁红绿测试：修复前 1.5 秒内收不到搜索完成信号；修复后约 0.9 秒返回，搜索线程仍存活；UNC/特殊字符只读 URI 与禁止写入均有回归覆盖。
- 受影响测试组：460 passed / 285.16s；完整测试集：872 passed / 336.04s。
- 最终全量测试后查询 Windows Application Error：Python / Qt / PPT Doctor 崩溃事件 0。
- 独立只读审查分为扫描搜索、PowerPoint 隔离、Qt 生命周期、功能分级四组；最终均为 Approve。数据库锁补丁另经 Claude 窄审，发现并闭环 UNC P1 后复审为 Approve。

## 已知安全边界

- 自动全盘扫描仍会遇到公司策略明确拒绝访问的目录；修复目标是“不误删、不假装扫完、给出提示”，不能绕过操作系统权限。
- 如果 PowerPoint COM 自身永久不返回，PPT Doctor 会暂停新预览，直到该隐藏进程退出或应用重启；这是刻意的安全取舍。自动终止握手完成的 Python 渲染子进程会留下独立的 PowerPoint 进程，因此禁止采用。
- 本批代码位于独立 worktree/功能分支；通过门禁不等于已经合并、打包或发布。
