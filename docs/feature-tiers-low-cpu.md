# 基础模式与低负载建库

## 原始目标

默认只做全盘 PPT 搜索与 PPT 统计；版本管理、Word/PDF 搜索和相似稿归组只给主动开启的高阶用户。任何建库状态下，搜索、按钮、翻页和窗口操作都必须保持响应。

## 已确认根因

1. 版本管理无条件启动 watcher、离线对账和版本库维护，即使用户只需要搜索。
2. Word/PDF 无条件进入全盘枚举和内容解析，扩大了首次建库工作量。
3. 每次索引变化后无条件重算全库 MinHash 相似稿，并为每个文件额外读取一次完整字节计算 SHA-256。
4. 首次/手动扫描最多启动八个普通优先级解析进程，容易与 PowerPoint 和桌面前台争抢 CPU、磁盘与内存。
5. 索引完成信号早于归组和数据库维护发出，主线程可能与仍在写库的 worker 竞争 SQLite 锁。
6. 主窗口启动逐像素处理 675×696 Logo，约 47 万次 Python 调用，实测单次约 0.84 秒。

## 实现决策

- 默认关闭：版本管理、Word/PDF 搜索、相似稿/重复稿智能归组。
- 基础索引只枚举 `.pptx/.ppt`；旧 Word/PDF 记录保留但不展示，重新开启后后台补建。
- 应用发起的建库固定使用低 CPU/I/O 优先级隔离进程，默认最多两个 worker；用户交互发生时暂停继续投递新任务。
- 进度区分“全盘发现文件”和“内容建库”，后者提供平滑速度与剩余时间；前者明确总数尚未确定。
- 相似稿关闭时不计算 SHA-256、不跑 MinHash、不读取旧分组；开启后缺少精确哈希的文件会逐步重建。
- 文件监听独立于版本管理：基础模式仍能实时更新 PPT 搜索索引，但不会生成版本快照。
- 所有完成通知必须在写事务和维护动作结束后发出。

## 验收红线

- 全新配置的三个高阶功能均为关闭。
- 基础模式的扫描、最近文件、状态统计和搜索均不泄露 Word/PDF。
- 建库时连续搜索、切换排序/筛选/设置、滚轮翻页，UI 事件循环最大额外间隔不超过 250ms；测试目标 p95 不超过 100ms。
- 应用扫描最多两个低优先级隔离 worker；交互期间不继续扩充解析队列。
- 版本关闭时不存在版本对账/维护线程；PPT 保存仍能触发实时索引。
- 取消或失败不得显示“建库完成”。
- 完整测试集、压力测试和独立评审通过后才允许合并主干。

## 最终验收

- 合成 A/B（300 份 PPT、3 次中位数）：基础模式相对高阶模式节省约 53% 建库墙钟时间、55% CPU 时间。
- 无并发 15 分钟最终代码高频压测：2,560 次操作，操作 p95 18.91ms、最大 75.38ms；999 次搜索全部完成，搜索 p95 12.07ms、错误 0；UI 独立监控最大间隙 79ms、慢间隙 0；末分钟 CPU 0%。
- 受影响测试 460 项、完整测试 872 项全部通过；最终测试窗口内 Windows Qt/Python 崩溃事件 0。
- 独立代码审查最终结论为 Approve；当前仅提交功能分支，不自动合并、打包或发布。
